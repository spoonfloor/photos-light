CURRENT DIAGNOSIS (verified in codebase)
============================================================================

1. Backend classifier is good; contract is incomplete

   db_health.py:
   - DBHealthReport has can_migrate, can_use_anyway, get_recommended_actions()
   - Classifies HEALTHY, MISSING, CORRUPTED, MISSING_COLUMNS, MIXED_SCHEMA, etc.

   app.py build_library_status_payload() (~4250):
   - Maps health → /api/library/status JSON
   - needs_migration payload includes missing_columns, can_continue
   - Does NOT expose can_migrate or recommended_actions

2. Startup path strands users

   Flow: init() → checkLibraryHealthAndInit() → GET /api/library/status
   static/js/main.js (~16119):
   - case 'needs_migration' → showCriticalErrorModal('db_needs_migration', ...)

   static/js/main.js showCriticalErrorModal (~5167):
   - db_needs_migration modal offers only "Open library" and "Reload"
   - Neither runs migration → infinite loop / stranded library

3. Switch path partially works; failure path is unwired

   app.py switch_library() (~5306):
   - Auto-calls migrate_db.check_and_migrate_schema on MISSING_COLUMNS/MIXED_SCHEMA
   - On failure returns 400 { status: 'needs_migration', action: 'migrate', ... }

   static/js/main.js switchToLibrary() (~15247):
   - Treats any !response.ok as generic throw — no handling for needs_migration
   - User sees toast/error, not migration recovery UI

   test_db_health_consistency.py:
   - test_switch_library_missing_columns_migrates_and_succeeds ✓ (happy path)
   - No test for startup-stranded needs_migration + user-initiated migrate
   - No test for switch failure → migrate action surfaced to UI

4. No dedicated migrate API for configured library

   migrate_db.check_and_migrate_schema() exists and is used inside switch_library.
   There is recover_library_database POST /api/library/recover-database for corrupt.
   There is NO POST /api/library/migrate (or equivalent) for the already-configured
   library path the user is trying to open on startup.

5. Schema tooling is current

   db_schema.py → v4 (date_added). migrate_db.py targets v4 columns/tables.
   Do not re-litigate schema version; wire the existing migrator correctly.

TARGET ARCHITECTURE
============================================================================

                    ┌─────────────────────┐
                    │   db_health.py      │
                    │  check_database_    │
                    │  health() → Report  │
                    └──────────┬──────────┘
                               │
                    ┌──────────▼──────────┐
                    │ build_library_      │
                    │ health_payload()    │  ← extend/rename from
                    │ (single translator) │    build_library_status_payload
                    └──────────┬──────────┘
                               │
         ┌─────────────────────┼─────────────────────┐
         │                     │                     │
  GET /api/library/status  POST /api/library/switch  POST /api/library/migrate (new)
         │                     │                     │
         └─────────────────────┼─────────────────────┘
                               │
                    ┌──────────▼──────────┐
                    │  Frontend action    │
                    │  router (main.js)   │
                    │  - critical modal   │
                    │  - switchToLibrary  │
                    └─────────────────────┘

Payload shape (minimum — keep backward compatible where tests depend on it):

  status: 'healthy' | 'needs_migration' | 'db_missing' | 'db_corrupted' | ...
  message: string
  library_path, db_path
  valid: bool
  recommended_actions: ['migrate', 'continue', ...]  ← from report.get_recommended_actions()
  can_migrate: bool                                ← from report.can_migrate
  action: 'migrate' | 'rebuild' | 'create_new'     ← primary recommended action (switch compat)
  missing_columns, extra_columns (when relevant)

Frontend action map (canonical):

  migrate   → POST /api/library/migrate → re-check status → load grid or show error
  rebuild   → startRebuildDatabase() (already exists for db_missing/corrupted modal)
  create_new→ existing create-library flow (switch missing DB — don't break)
  continue  → only if product explicitly supports degraded open (can_continue today);
              if unsupported, omit from UI rather than ship a broken path
  abort     → Open library (picker) / first-run shell

Critical-error modal for db_needs_migration should mirror db_missing pattern:
  Primary:   "Update database" (migrate)
  Secondary: "Open library" (picker)
  Remove or demote useless "Reload" unless it re-runs migrate (prefer explicit migrate).

SAFETY (migration-specific)
============================================================================

Before first write to an existing DB in /api/library/migrate:
- Copy DB to .library/backups/ (pattern may exist — grep backup, DB_BACKUP_DIR).
- If no backup helper exists, add minimal copy-to-backups in library_layout or
  reuse existing recovery/quarantine patterns; do not skip backup on user-initiated migrate.

migrate_db.check_and_migrate_schema currently prints to stdout; for API use consider
returning structured success/failure (extend function or wrap) so the endpoint can
 distinguish partial failure vs complete.

Never DELETE or overwrite user media files during schema migration.
